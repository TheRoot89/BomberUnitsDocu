14 About concepts
15 ----------------------
16 
17 Have at look at the diagram below: the first step in development is getting an idea of your game design, aka.
18 what features you want to have in there from a top-level or user experience view.<br>
19 From there on you need to break down the requirements taking into account the limitations of your system on
20 the SW level, until you reach your SW solution.<br>
21 
22 ~~~plantuml
23 @startuml
24 
25 hide empty description
26 
27 state "Game Design" as GameDesign
28 state "System Requirements" as SystemRequirements : "We need an animated main menu!"
29 state "System Solution" as SystemSolution : "Use the Animator to shift UI elements"
30 state "SW Requirements" as SWRequirements : "Allow easy composition of elements and pages ..."
31 state "SW Solution" as SWSolution : "The following classes are created, also see the How-To"
32 state "Architecture Docu Pages" as Arch
33 state "Generated API Docu" as GenAPI
34 state "How-To Pages" as HowTos
35 
36 GameDesign -> SystemRequirements
37 SystemRequirements --> SystemSolution
38 SystemSolution --> SWRequirements
39 SWRequirements --> SWSolution
40 SWSolution --> Arch
41 SWSolution --> GenAPI
42 SWSolution --> HowTos
43 
44 @enduml
45 ~~~
46 
47 Each **Concepts** page thus explains:
48 
49 * **System Requirements**: This states a feature from the Game Design step in more detail.
50  Also see: \ref page_game_design
51 * **System solution**: The approach taken to realize the above feature from a top-level perspective.
52 * **SW requirements (in part)**: Constraints on how to implement the system solution above.
53 * **SW solution**: Explanations on how the solution works in SW. We typically have:
54  * The concept of how the SW architecture works for this.
55  * The detailed sw design docu as code comments, generated into the Doxygen API docu.
56  * Links to How-Tos for using the feature.
