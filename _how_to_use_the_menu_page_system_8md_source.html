3 How to use the Menu Page System
4 ===============================
5 
6 This is a How-To for the concept of \ref page_ui_menu_system, with components explained on its architecture
7 site here: \ref page_ui_menu_page_system.
8 
9 Please note this concept is responsible for hiding or showing different parts of a UI. It can also be used to
10 implement actual menu pages. The switching of pages could be done by utilizing a state machine approach.
11 In BomberUnits, the main menu has its own state machine, with each state corresponding to a menu page as
12 introduced here.
13 
14 > There is a test scene to look at with this How-To: Assets/Test/MenuAnimationWithLayoutsTest.unity
15 
16 How to set up menu pages.
17 -------------------------
18 
19 1. Have some object, such as a `UIManager` to organize the logics between page transitions. It decides *which*
20  menu components should switch *when*, using the provided callbacks. This logic is however not part of this
21  How-To. Just be aware that you will need to register function callbacks to your buttons which can trigger
22  page switches directly or activate some other mechanism, such as a full menu state machine.<br>
23  See: \todo Link to Menu state machine docu needed.
24 
25 2. Do the usual Layout with the Unity System:
26  - Versions of *Layout Group* components are added to the parents of groups of objects to be layed out.
27  - Think about using *Layout Element* components for these elements to be able to set minimum dimensions.
28  This way eventual parent Layout groups of this element do not change too much when its content is moved
29  out and disabled by our animations.
30  - Example in the test scene: `MainHLayout`
31 3. Elements you want to animate and which are part of a Layout group need an empty and invisible container
32  object, which is only a Rect Transform. This way the element can be panned out while still being a part of
33  the containing layout.
34  - Example in the test scene: `Element3LayoutContainer`
35 4. The elements itself, such as `Element3`, need the `Animator` and `MenuElement` components.
36 5. Give the `Animator` component the `MenuElementTransition` controller asset.
37 6. Assign the container object to the `MenuElement`'s *Container* field if any.
38  This is needed so the container can be disabled when the element is disabled. This will change the
39  organization of the enclosing *Layout Group*, so don't give a reference to the container if you don't want
40  it to be disabled when the element is hidden (see in the example: container setting is empty).
41 7. Assign groups of elements to animate together to the "Page Elements" fields of empty objects with the
42  `MenuPage` component. These can represent menu pages but don't necessarily have to.
43  You should group your logical pages under its own parent, e.g. *Pages*.
44  > Hint: The page inspector provides a toggle to show or hide the page GameObject while editing, which
45  should be handy.
46 8. Handle your menu transitions:
47  - Some convenience functions are available for pages and elements directly, so you can assign these to
48  your button's callbacks directly, without doing an extra UI-manager-detour.
49  - Or add a number of triggers to the corresponding setting for your pages in the editor. This will take a
50  button and a command each, which will be set to the button automatically once the page is shown.
51  \todo Trigger concept not final, as too specific for main menu state machine!
