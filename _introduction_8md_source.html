100 ### 5. An example of a typical SW change workflow
101 For a feature implementation, things may go like this:
102 
103 1. We decide together which features we want to implement next and move these on our
104  [GitHub Repo Projects page](https://github.com/TheRoot89/BomberUnits/projects)
105  from the "Ideas" column to the "ToDo" column.
106 2. Choose one of these tasks and convert the note card to a full *Issue* via "..." -> "Convert to Issue", if
107  that was not already done before. You also assign yourself as the "Assignee" of that task.
108 3. Fill the issue with whatever info you feel necessary or start a conversation using the comment section.
109 4. Create a branch (from master most of the time), mentioning the issue number in its naming.<br>
110  Example: "feature/#4_implement_multiplayer".<br>
111  Using the "feature/" prefix allows us to sort the branch names after features, bugfixes, releases and
112  whatever categories we want to define.
113 5. Work on you branch and add the feature, using Git and pushing your changes to our repo. Best move your
114  issue from the "ToDo" column to the "In progress" column.
115 6. If there were changes on the master branch from when you branched off it, merge "master" to your branch to
116  check compatibility, resolve merge conflicts and test whether everything still works as expected.
117 7. Go to GitHub, select your branch on the left and click "New pull request". Fill in the details of what you
118  did on your branch, assign Reviewers on the top right and click "Create pull request".
119 8. Once the reviews from us approve your changes, the created pull request (PR) can be merged in GitHub.
120  If you don't need it any longer, your branch can be deleted (this will only delete the name, not the
121  commits you did) and you can close your Issue and move it to the "Done" column! Yeah!

126 On Unity
127 --------
128 Unity is a game engine and tool collection for this engine. There is no real winner when comparing Unity to
129 other free engines, like Unreal - so we just start with the one that looks easier to pick up.
130 
131 Unity comes with its own graphical editor to arrange your game world in what Unity calls *Scenes*. In-game you
132 as a player will always be in one of these Scenes.
133 It may and will feature so-called *GameObjects*, representing any kind of entity present int the Scene, e.g.
134 a player figure, bullets, particle effects or invisible objects only there to hold *Scripts*.
135 
136 Scripts represented the logic of a game, implemented in C#. These can be written as in any other project,
137 defining and creating objects and calling the Unity API to act on the game world.
138 
139 Interaction of your Scripts with the engine comes in two directions:
140 
141 * Script to engine: Your Script causes something to happen by calling the API, e.g. to change the position of
142  a game object.
143 * Engine to Script: Some predefined Script functions can be called from the game and are thus called
144  *Callbacks*. An example is the *Update()* function called by the game for every frame.<br>
145  In order for this to work your Script's class has to be a *Behavior*, by inheriting from the *MonoBehavior*
146  class.
147  In order for your class to be instantiated and the Callbacks (like *Update*) to be called, it also has to be
148  added to a *GameObject* in the Unity editor. That *GameObject* will now have the additional *Behavior* you
149  code for it.
150 
151 > Example: A *PlayerController* script is often added to the player's figure in the game world. Each frame its
152 > *Update* function will be called by the engine. Within that function, code may be added to check the input
153 > keys a player may be pressing to control his figure and the figure may then be moved accordingly.

155 About the game
156 --------------
157 So, **Bomberman** is a classic game type (Google!). We first want to replicate its classic, simple scheme of
158 moving your player figure, lay bombs, collect PowerUps and finally explode your opponents.
159 
160 As this is vitally important today, we also want to support **mixed multiplayer** (local, online).
161 Of course a game does not just start directly with a level and your fellow players already there.
162 There will have to be a menu UI, (online) player management, level selection, game-over screen etc...
163 
164 When this runs stable, we want to add our own take on the genre and add our **own features**. This even more
165 so than the above points is a group effort, where we can brainstorm, decide, implement and test together.
166

186 ### Game Design
187 The core of any project: figuring out what the f*** we actually want. In order to get going, we need at least
188 some basic ideas, preferably with an interesting twist. Then we need to refine these ideas to answer questions
189 such as:
190 
191 * How will playing this be fun?
192 * What are the core elements and themes?
193 * What shall the Player's experience be like?
194 * What should the graphical style be like? What that of the UI?
195 * What main functionality will the SW have to provide?
196 * What other, crazy ideas do we have? Do they fit in?
197 
198 The **tools to support this** are up to discussion - I just started with plain Powerpoint.
199 
200 ### Planning & Teamwork
201 So, imagine we have quite a big list of tasks, big and small, crazy ideas and stuff we just want to try to
202 see whether they work. We need to:
203 
204 1. Decide what has to be done in which order from a technical perspective.
205 2. What we want to do first of the things that require no special order. What are the ratios of gain vs.
206  effort for the features we want to add?
207 3. Who wants to do what? Is everyone happy with what he is doing and where we are going?
208  Does anyone need help?
209 
210 Also, stuff needs to be taken care of:
211 
212 * When and how do we meet?
213 * How do we want to collaborate? Are there any rules?
214 * Most importantly: How can we improve?
215 
216 One **tool to support this** is surely the project tab and issue tracking of
217 [GitHub](https://github.com/TheRoot89/BomberUnits), where our code is hosted anyway. It offers a way to
218 collect, sort and prioritize ideas as notes, track tasks, and rant about them in strong language.
219 Information not related to ideas or tasks may be tracked using these nice Markdown pages.
220 
221 ### Art Design & Modelling
222 If this is not a text adventure, we will need to show some art, be it 2D or 3D. How can we achieve this
223 without hiring Weta Workshop? How do we achieve a reasonable look with comparably low effort?
224 
225 The **tools to support this** are manyfold, with many specialized solutions for the different artsy tasks,
226 depending on what we want. Simple models can be made for example with [AssetForge](https://assetforge.io/),
227 while arbitrarily complex stuff can be done with Blender. Maps and textures can be found, also free, all over
228 the web. The final look will also largely depend on the Postprocessing Stack, allowing cool looks for low-poly
229 models.
230 
231 ### Rigging & Animation
232 Rigging is the task of providing a virtual skeleton, or *rig*, for your models. This allows to deform the
233 models to assume natural looking poses. A sequence of such poses forms an Animation, which is a kind of asset
234 importable and replay-able in Unity.
235 
236 **Tools to support this:** Rigging is typically supported by modeling SW, such as Blender.
237 Animations may be done in the Unity editor as well, while the final chaining of animations with animation
238 state machines is a task of game development, also supported graphically by the Unity Editor.
239 
240 ### Level Design
241 This is what the title implies, and you have probably seen good and bad levels abound yourselves. It is a bit
242 of a cross-over task:<br>
243 The design aspect of this is a part of the *Game Design*, especially when thinking of cool level features,
244 which will need support from a *Scripting* perspective.
245 
246 For the actual modeling of a level Scene, using
247 mainly the Unity Editor as **Tool**, you will combine Models as well as maybe create your own stuff, as a
248 *Modeling* guy would do. Poor fella.
249 
250 ### Scripting
251 This is the art of learning what the Unity API has to offer, how to use that to design an understandable and
252 reusable SW architecture and - of course - coding the thing in C# (or Java Script alternatively). The scripts
253 bring the game objects to life, let you play with friends, produce tons of log output, and so on...<br>
254 Game development may sound like it is all about Coding/Scripting, but as you see, there are many daunting
255 tasks each worthy ofa whole profession in the industry.
256 
257 The **tools to do that** are Unity and a supported IDE, such as Visual Studio or Visual Studio Code. Have a
258 look at the Tooling section (\ref page_tooling) for how to set those up. There is no black magic involved, as
259 Unity pretty much has the whole toolchain, up to deployment, built-in. Nice.
260 
261 ### Effects
262 This category wraps up some of the typical features found in Game Engines: particle systems, sound effects and
263 the relevant scripting making effects *affecting* the game world. This is also a bit of a crossover task, and
264 you get to play with explosions. Also in the effects category are screen effects, supported by the rendering
265 pipeline used, which is quite the topic of its own.
266 
267 The Unity editor may be all the **tool** you need to get started. Also Unity gives away quite a lot of
268 free-to-use effects we can start with. Building on that, you may need to roam the Net and use some tools of
269 the other disciplines to create you very own uber-effects!
270 
271 ### Post processing
272 In Post processing, meaning after the basic rendering of the frame, it is possible to apply further
273 transformations to apply effect or improve style and image quality. Typical tasks are Anti-Aliasing or the
274 Bloom effect. This has to be designed, tuned and tested.
275 From the **tooling** perspective, all this should be supported by Unity itself I think - but I am by no means
276 experienced in that area.
277 
278 ### DevOps
279 If we decide to have, e.g. build automation, common tool configurations (such as for Doxygen in the doc
280 folder), etc. - someone will have to look after that. Someone willing to be the all-important enabler of the
281 whole team. That someone could be you!
282 
283 The skills learned/needed here are probably no special to game development, and the **tools** are manyfold,
284 depending on what the team deems worthy automating.
285 
286 ### Testing
287 No time for the other stuff? No worries - we need people that from time to time have a look at the current
288 state and tell us what we f**** up. At pat on the back is also helpful sometimes.
289 
290 And then, there is *proper* testing, including *Unit Tests*. It may not seem reasonable for the dynamic nature
291 of game scripting code, but it is very well possible and supported by Unity's own set own **tools**.
292 The scripters will kiss the feet of the guy who tells them how to use those tools. At the very latest when
293 complexity increases and devs become afraid of breaking things accidentally.
