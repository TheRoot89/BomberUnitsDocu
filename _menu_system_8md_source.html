3 The menu page system
4 =================================
5 
6 The menu page system is intended to provide animations for menu components as well as grouping, but still
7 preserve the Unity Layout features etc.
8 The classic approach is having multiple full menu "pages" readily set up with all their children on the canvas
9 at the same time. These would then be made visible or invisible as pages are switched. We instead want to be
10 able to animate and pan into view the individual elements (often panels with buttons and images)
11 independently. By parenting, also full pages can be animated with this of course.
12 
13 Basics
14 -------
15 An animation is saved as its own file (*.anim), which describes the transition of controlled parameters(e.g.
16 its position) over time. Each animation can be viewed as a state an object is in with transitions between
17 different animations. These transitions can be done abruptly or smoothly, which is what we use here.
18 
19 The *Animator* component is added to an object that shall be animated. It gets assigned an
20 *Animation Controller* to organize in a state-machine fashion when which series of animations is applied to an
21 object.
22 Each animated object has its own animation state machine it runs through. The Animation Controller also
23 defines how the different animations aka *states* transition (*blend over*) into each other.
24 
25 The System
26 -------------
27 Have a look at the test scene: Assets/Scenes/Tests/MenuAnimationWithLayoutsTest.unity
28 
29 - We have 5 animations: one for each state of being *hidden* to each of the four sides of the screen, and one
30  for being centered aka. *shown* on the canvas (i.e. a position of (0,0,0) ).
31 - Each "animation" only consists of one frame: the new position of the object shall have. The actual visible
32  animation of an element sliding out or in is actually the visible transition between those
33  1-frame-animations.
34 - Each item to be animated also needs the `UIMenuElement` component, which features the scripting to start the
35  animations and provides necessary settings.
36 - UIMenuElements can be grouped to form a `UIMenuPage`, by listing these elements on an empty object featuring
37  the `UIMenuPage` component. They do not have to be children of the page, but only be listed there! This
38  allows for reuse of elements on multiple pages.
39 - The pages can be children of a 'Pages' parent but don't have to be.
40 
41 *Elements* and *Pages* provide functions to hide or show them. Have a look at their documentation for details.

47 Coping with Layouts
48 -------------------
49 Layouts, together with anchors, are very useful calculating the correct sizing and positioning for all objects
50 automatically.
51 But Layouts also prevent objects from moving, i.e. being animated with a shift because they have control over
52 the position (the coordinates are greyed-out). As a solution, we need to layout an invisible parent of the
53 object-to-be-animated and shift the actual element relative to its parent.
54 
55 So, to get both worlds, set the the content you want to animate as child of an empty RectTransform object.
56 This *container* will get layed out by its respective parent having a layout component. The actual
57 `UIMenuElement` in its shown state is set to be in relative position (0,0) and is thus the same as its
58 container. This relative position can be modified by the Animator to move off-screen, while the container
59 actually stays in place as set by the layout.

61 How to set it up
62 -----------------
63 
64 1. Have some object, such as a `UIManager` to organize the logics between page transitions. It decides *which*
65  menu components should switch *when*. Also menu button setup etc. has to be done here when switching
66  between pages.
67 2. Do the usual Layout with the Unity System:
68  - Versions of *Layout Group* components are added to the parents of groups of objects to be layed out.
69  - Think about using *Layout Element* components for these elements to be able to set minimum dimensions.
70  This way eventual parent Layout groups of this element do not change to much when it's content is moved
71  out and disabled by our animations.
72  - Example in the test scene: `MainHLayout`
73 3. Elements you want to animate and which are part of a Layout group need an empty and invisible container
74  object, which is only a Rect Transform.
75  - Example in the test scene: `Element3LayoutContainer`
76 4. The elements itself, such as `Element3`, need the `Animator` and `UIMenuElement` components.
77 5. Give the `Animator` component the `MenuElementTransition` Controller.
78 6. Assign the container object to the `UIMenuElement`'s *Container* field if any.
79  This is needed so the container can be disabled when the element is disabled. This will change the
80  organization of the enclosing *Layout Group*, so don't give a reference to the container if you don't want
81  it to be disabled when the element is hidden.
82 7. Assign groups of elements to animate together to the "Page Elements" fields of empty objects with the
83  `UIMenuPage` component. These represent your menu pages and can be used in your UI manager's button
84  callbacks to call to switch pages.
85  You should group your logical pages under its own parent, e.g. *Pages*.
86  - Hint: The page inspector provides a toggle to show or hide the page GameObject while editing, which
87  should be handy.
88 8. Handle your menu transitions with your own button callbacks on your UI Manager.
89  Some convenience functions are also available for pages and elements directly, so you can assign these to
90  your button's callbacks directly, without doing an extra UI-manager-detour.
