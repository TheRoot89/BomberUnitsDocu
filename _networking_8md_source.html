3 Networking
4 ==========
5 
6 This is a big topic, so it is divided up:
7 
8 Some Definitions
9 ----------------
10 
11 * *Player: Game representation of one physical human or bot player. Has a player slot in menus and a player
12  figure with controller in a match. Each Player can be customized which is synced over the net.
13  There is always at least one (Main) Player, which cannot be removed, while other local players can be added
14  and removed freely.
15 * *Client*: An instance of the game, which can play offline, host or join a match.
16 * *Main Player*: Each Client needs to have at least one human Player, which is set up and loaded during
17  startup and cannot be removed. This is the Main Player.
18 * *MasterClient*: The instance of the game having control over a match. In an online game, the Master is the
19  owner of the current room.
20  In an offline game, the current client is considered MasterClient as well (over the local offline room).
21  He is still also a Client.
22 * *JoinedClient*: All clients in a room, which are not the MasterClient. They joined the room afterwards.
23 * *Room*: Association of one or more Clients as a group playing together. This allows data exchange,
24  instantiation, concurrent scene loading and more features. Being in a room is the precondition for Photon's
25  in-game API to function.
26 
27 Offline Play
28 ------------
29 
30 After the startup, the local client is not connected automatically. Offline Play is enable by simulating a
31 connection to the servers via Photon's so-called 'Offline Mode'.
32 Local players can be added and a local match can be started. The goal is to use as much code as possible for
33 offline and online play.
34 Instantiation happens - internally - as usual for offline games, as opposed to online instantiation over the
35 network.
36 In Photon Offline mode, a 'room' for this single client is created as well, allowing use of the same data
37 storage (etc.) functionality which would be used during online play.
38 
39 Connecting
40 ----------
41 
42 A connection attempt explicitly ends the simulated 'Offline Mode'. This means data shared via room properties
43 has to be saved locally as well and is to be carried over to a new 'online room'.
44 A connection attempt can fail for a number of reasons, including user or network interruption.
45 See 'Disconnecting' for an overview of this topic.
46 
47 There are multiple successive steps involved:
48 
49 1. Connecting to the Photon Master Server
50 2. Creation of a new room
51 3. Joining the new room as MasterClient
52 
53 Hosting (aka. being the MasterClient)
54 -------------------------------------
55 
56 Upon successful connection, the local client acts as the MasterClient of the new room with only him present.
57 Being the MasterClient implies having authority over a number of networking aspects, such as:
58 
59 * Extended user options as Host of the game (Level selection etc.)
60 * Scene transitions
61 * Networked object Instantiation and Destruction
62 * Match Start, End etc. (Initiation of game state transitions)
63 
64 Joining (aka. being a JoinedClient)
65 -----------------------------------
66 
67 Joining a game means selecting one of the available hosted rooms. In joining (also as MasterClient), a Client
68 sends information on his players to all other clients in the room. Each client thus keeps track of all
69 players.
70 \todo: This is to be changed by using room properties (see below)!
71 A JoinedClient has restricted authority over certain game aspects, see MasterClient section.
72 
73 Disconnecting
74 -------------
75 
76 A disconnect comes in two shapes:
77 
78 * Actual disconnect from the Internet game service -> majority of cases.
79 * Logical disconnect from the simulated connection to the Master Server during Offline Mode.
80 
81 While the second form should ideally never happen, we should guard against it.
82 In any case, the requirement after such an event is to:
83 
84 * Display a message stating the reason to the user
85 * Regain a stable state in activated Offline Mode, allowing local play at least. The game state must be reset
86  so that no side effects remain.
87 
88 This requires:
89 
90 * Detection of a Disconnect: Done via an Event from the NetManager
91 * Halting of current gameplay
92 * Restriction of user interaction with Menus & Game Objects: Display a Message Overlay
93 * Reset of the game state towards the main menu in the background, including re-activation of Offline Mode.

100 Similarities between Host and joined Clients
101 --------------------------------------------
102 
103 This is an important part from an implementation perspective:
104 
105 * Local players and remote Players are displayed in Slots, with the local players coming first and displayed
106  differently than remote ones.
107 * An "add Player" button is displayed if new players can still be added. The maximum amount of local and total
108  players thus has to be the same across Clients.
109 
110 Networked game aspects
111 ----------------------
112 
113 There are two main points for what and how to synchronize aspects of the game over the network.
114 
115 1. Consistency of the game state over all clients (Sync): Avoid e.g. a player being hit on one client but not
116  on the other.
117 2. User experience: The game should not offer advantages or disadvantages for hosts or clients introduced by
118  lag or deviating not-networked game object states.

125 Other Networking Tasks
126 ----------------------
127 
128 * Player Management: Changes to players need to be synced. This is best done via specific events:
129  This encompasses all player information valid independent of match states. Send all local players if:
130  * Entering a new room -> send to all, unbuffered (bit of overhead for self, as it creates an update)
131  * A new client enters a room -> send to him specifically
132  Send one local player if:
133  * A new local player is added -> send to all others
134  * A local player is changed -> send to all others
135  * A local player is removed -> (own event) send to all others
136 * Player Figures: information concerning the state within a match
137  * Player figure instantiation -> information on spawn positions is sent out by the match manager.
138  Then the spawning is done by each client separately. Additional information on the looks of
139  every figure is read from the player list by each client and applied during or after instantiation.
140  * Player figure transform -> Photon Transform View
141  * Hitpoints -> monitor and sync for UI via PhotonView
142  * Bombs available -> monitor and sync for UI via PhotonView
143  * Status effects -> monitor and sync for UI via PhotonView
144  (No status effects impl. yet! Could also be done by events.)
145  * Being Hit -> calculated on owning client for correct user experience, propagation done via hit points
146  * Death -> calculated on client, propagation via event. (Could also be done via hit points client-side,
147  but those are shared with unreliable transmission, which is not good for something that important.)
148 * Game objects: in-game objects with influence on game state or UX.
149  * Bomb instantiation -> Bomb placement sent via RPC and instantiated per client. RPC used to target the
150  player figure instance setting the bomb individually and correctly.
151  * Bomb explosion -> Calculation per client with Hits on Player figures and environment. This is reliable
152  enough to be calculated client-side. The explosion timing with independent instantiation needs to be
153  tested for potential negative influence. A global explosion target time could be sent with the RPC or
154  the delay calculated and subtracted from the timer.
155  * Power-Up Spawn: The type is random and thus decided on the master only and spawned for all -> photon
156  view and Photon-based instantiation needed. No sync of transform needed as animation not relevant for
157  gameplay.
158  * Power-Up Collection: Has to be calculated on client-side for UX. Destroy via network.
159 
160 Photon Networking mechanisms
161 ----------------------------
162 
163 * RPCs: Call a specific function on all Client's PhotonViews with the same ID. You can transmit function
164  parameters as well. This is good for keeping the same objects in sync.
165  -> Use to act on objects that have a PhotonView anyway.
166 * Events: RaiseEvent allows to send an event package with data to specific clients. Any method can be
167  registered to handle this event. No PhotonView is needed. This is good to announce ...err... events,
168  concerning multiple receivers and can be targeted better than the RPC.
