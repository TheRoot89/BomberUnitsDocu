3 BomberUnits
4 ======================================
5 
6 BomberUnits is a Bomberman clone in Unity as a learning project. It is developed in 3D and supports mixed
7 local and remote multiplayer via [Photon](https://www.photonengine.com/en-us/Photon).
8

15 **How to start:** Check out the correct commit, install and assign the correct Unity version to the project in
16  UnityHub, load the project, load and play the "MainMenuScene".

27 About the Documentation
28 -----------------------
29 
30 This documentation is based on a combination of Doxygen and Markdown. While the Markdown part is (mostly)
31 readable directly or on GitHub, better go to our "GitHub Pages" for the fully generated docu. It will be
32 updated from the master branch from time to time:
33 
34 > [https://theroot89.github.io/BomberUnitsDocu/index.html](https://theroot89.github.io/BomberUnitsDocu/index.html)
35 
36 The docu shall be built up like this:
37 
38 ~~~plantuml
39 @startuml
40 
41 hide empty description
42 
43 state "Game Design" as GameDesign
44 state "System Requirements" as SystemRequirements : "We need an animated main menu!"
45 state "System Solution" as SystemSolution : "Use the Animator to shift UI elements"
46 state "SW Requirements" as SWRequirements : "Allow easy composition of elements and pages ..."
47 state "SW Solution" as SWSolution : Code explanations (classes "UIMenuPage" etc.) and Usage Info
48 
49 GameDesign -> SystemRequirements
50 SystemRequirements --> SystemSolution
51 SystemSolution --> SWRequirements
52 SWRequirements --> SWSolution
53 
54 @enduml
55 ~~~
56 
57 As mentioned above, the docu consists of auto-generated content from the parsed scripts and additional pages
58 on various topics.
59 Most importantly, the **Concepts** pages each explain: (See: \ref page_concepts)
60 
61 * **System Requirements**: A typical problem a game program such as this poses, which needs to be solved.<br>
62  In essence, these are the results of the Game Design process, i.e. what we want to
63  achieve. The GameDesign can currently be found in a ppt file.
64 * **System solution**: The approach taken to solve the problem from a top-level perspective.
65 * **SW requirements (in part)**: Constraints on how to implement the system solution best.
66 * **SW solution**: Explanations on how the solution works if solved by multiple classes. Also find
67  **usage information** here for how the implementation is to be used.<br>
68  The actual **SW solution detailed design** is represented by the code comments in the
69  implementation files. This is parsed and accessible at the Doxygen output for each of the
70  related classes.
71 
72 These documentation pages are viewable as text files, rendered as Markdown, e.g. in VSCode preview, or after
73 generating the Doxygen docu HTML as explained below.
74 
75 The Doxygen-specific "\\ref" links and commands will not work when viewing the files directly or as Markdown,
76 but you may search textually for the referenced unique keywords to find the relevant content.

78 Building the Documentation locally
79 ----------------------------------
80 Prerequisites:<br>
81 You may have to install Doxygen and GraphViz (dot) from the internet. Google is your friend here.
82 
83 Use the PowerShell script ```build_docu.ps1``` in the *Doc* folder to build the documentation.
84 It calls ```doxygen Doxyfile``` to call doxygen with its configuration and creates a link to the start page.
85 <br>
86 Double-click it to execute and select PowerShell as program to open it with if necessary. In order to allow
87 PowerShell script execution you may have to execute the following in a PowerShell you started with Admin
88 rights: ```Set-ExecutionPolicy Unrestricted```
