3 SW Architecture
4 =================
5 
6 These pages shall give an overview on how the overall system works. The SW architecture realizes the game
7 concepts (See \ref page_concepts) in a hopefully understandable way. Below the architecture come the
8 individual SW components, which are documented by code comments, generated into the API.

14 Overview
15 --------
16

51 
52 The **Game Management** Layer is responsible for various basic, background tasks. Its components are
53 implemented as behaviours and attached to invisible management GameObjects named after them.<br>
54 Those components which track their state have an additional state machine script attached to the same object.
55 
56 Please find information on the individual managers in their source comments or the detailed description
57 section in their Doxygen docu here:
58 
59 | Component | Short description |
60 |-------------------------------|-------------------------------------------------------------------|
61 | BomberUnits::GameManager | Manages the overall game state: startup, loading, quit |
62 | BomberUnits::UIManager | Manages UI display and inputs and delegates reactions accordingly |
63 | BomberUnits::MatchManager | Initializes and manages a single match's state |
64 | BomberUnits::PlayerManager | Manages local and remote players |
65 | BomberUnits::NetManager | Manages Network interactions with Photon for enabling MP |

73 Custom Classes
74 --------------
75 Additionally, there is a range of custom classes which don't need to be behaviors. Most importantly:
76 
77 * Container types, such as:
78  * BomberUnits::Player
79  * BomberUnits::InputScheme
80  * BomberUnits::GameSettings
81  * BomberUnits::Level
82 * UI classes for managing UI components, grouping and animation. See:
83  * BomberUnits::UIMenuPage
84  * BomberUnits::UIMenuElement
85  * BomberUnits::UITrigger
86 
87 Other Scripts
88 -------------
89 Categorized into folders we also have scripts for:
90 
91 * *Game Mechanics*:
92  * BomberUnits::PlayerController : manages player interaction with the player figure
93 * *Attributes*, which provide further functionality/interactions to GameObjects. Each attribute script can be
94  configured from the editor, or by loading from the settings.
95  * BomberUnits::Breakable : Exposes the Break() to replace a model by its broken variant.
96  * BomberUnits::Explodeable : Exposes the Light() function to trigger a delayed explosion of the object.
97  * BomberUnits::LootSpawner : Exposes the DropLoot() function
98 * *UI Behaviours*, providing UI features for objects they are attached to:
99  * BomberUnits::UILabel
100 * *Misc*, for other topics, such reading and writing from/to files.
101 * *State Machines*, which are associated to certain objects:
102  * BomberUnits::GameStateMachine
103  * BomberUnits::MainMenuStateMachine
