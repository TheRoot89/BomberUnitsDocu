3 SW Architecture
4 =================
5 
6 These pages shall give an overview on how the overall system works. The SW architecture realizes the game
7 concepts (See \ref page_concepts) in a hopefully understandable way. Below the architecture come the
8 individual SW components, which are documented by code comments, generated into the API.

77 Management
78 ----------
79 
80 The **Management** Layer (click for details) is responsible for various basic, background tasks.
81 The following important managers exist, with each topic being fairly independent.
82 Have a look at their detailed design docu:
83 
84 | Component | Short description |
85 |---------------------------|------------------------------------------------------------------|
86 | Management::GameManager | Manages the overall game state: startup, loading, quit |
87 | Management::UIManager | Manages UI display and inputs and delegates reactions accordingly|
88 | Management::MatchManager | Initializes and manages a single match's state |
89 | Management::PlayerManager | Manages local and remote players |
90 | Management::NetManager | Manages Network interactions with Photon for enabling MP |
91 
92 Most important is the **GameManager**, featuring its own state machine:
93 
94 * Concept: \ref page_gameManagement_controlFlow
95 * Concept: \ref page_handling_state
96 * State Machine: Management::GameStateMachine
97 * Also see: \subpage page_game_startup : How the components work together during startup
98 
99 Additionally, there is a range of custom classes which don't need to be behaviors. These are often data
100 structures, such as:
101 
102 * Management::Player
103 * Management::InputScheme
104 * Management::GameSettings
105 * Management::Level

107 UI
108 ----
109 
110 The UI implementations one the one hand need to provide functionality to compose, animate and switch menu
111 pages.<br>
112 Architecture Details: \subpage page_ui_menu_page_system <br>
113 Concept page: \ref page_ui_menu_system <br>
114 
115 | Component | Short description |
116 |---------------------------|-----------------------------------------------------------|
117 | UI::MenuPage | Groups MenuElements to be shown together as a menu page |
118 | UI::MenuElement | Added to game objects to enable their animation |
119 | UI::Trigger | Provides Button Callback registering at creation |
120 | UI::MainMenuStateMachine | Treats each main menu page as a state |
121 
122 On the other hand there are single-use pop-ups (like simple menus without transitions).
123 These will be called Dialogs, but this is work in progress.
124 
125 And finally additional UI content, such as in-game hovering labels and other effects:
126 
127 * UI::Label
128 * UI::Billboard

130 Mechanics
131 -------------
132 
133 Game Mechanics represent core game design ideas, spread out over various scripts. Some of these provide
134 input management (Mechanics::PlayerController), while other provide behaviours for game objects:
135 
136 * Mechanics::Breakable : Exposes the Break() to replace a model by its broken variant.
137 * Mechanics::Explodeable : Exposes the Light() function to trigger a delayed explosion of the object.
138 * Mechanics::LootSpawner : Exposes the DropLoot() function
