3 BomberUnits
4 ======================================
5 
6 ![Bam! (Pre-alpha footage)](/doxy_in/images/Screenshot1.png)
7 
8 BomberUnits is a Bomberman clone in Unity as a learning project. It is developed in 3D and supports mixed
9 local and remote multiplayer via [Photon](https://www.photonengine.com/en-us/Photon).

76 \dir Assets
77 \brief Contains all actual resources for the game, also visible in the Unity Editor Asset Browser.
78 \details This encompasses in particular the scripts, as well as binary resources such as images, sounds,
79  model files etc.
80 
81 \dir Assets/Scripts
82 \brief Contains all C# scripts, sometimes together with closely linked resources.
83 \details Having resources next to the scripts should be an exception, which sometimes makes sense due to the
84  close coupling of script and required other files, such as the state machine configuration for a
85  state machine class.
86  The only other location where scripts are held is the 'Test' folder, which is kept separate from the
87  production code, together with all relevant resources.
88 
89 \dir Assets/Scripts/Editor
90 \brief Scripts expanding the Unity Editor with add. functionality have to be placed inside an 'Editor'
91  folder. See namespace \ref editor.
92 \details These scripts will be discovered and executed for the editor automatically.
93  There are different types of editor scripts:
94  * Introduction of new Labels, like [ReadOnly].
95  * So called property drawers, which influence the Inspector appearance of specific classes.
96  * Additional windows or menu entries which will show up automatically in the menu at the top.
97  This currently encompasses two scripts under the 'Window' category, which help in finding missing
98  links in the Editor, e.g. after merges etc.
99 
100 \dir Assets/Scripts/GameManagement
101 \brief This category includes scripts related to the general functionalities of the game.
102  See namespace \ref management.
103 \details Most importantly this includes the main GameManager component. See sub-folders for others.
104  More scripts are moved to their own namespaces when the need arises.
105 
106 \dir Assets/Scripts/GameManagement/Level
107 \brief Includes definitions of level properties and the logics for discovery, selection, handling, etc.
108  (No own namespace yet.)
109 \details Level Management is wip! One important addition would be the representation of the level in-game,
110  instead of having this in the MatchManager!
111 
112 \dir Assets/Scripts/GameManagement/Match
113 \brief Includes scripts related to managing a single match, i.e. init, startup, pause, end and other logics.
114  (No own namespace yet.)
115 \details Match Management is currently in a draft state.
116 
117 \dir Assets/Scripts/GameManagement/Mechanics
118 \brief Encompasses mix-and-match scripts for making the game mechanics come to life.
119  See namespace \ref mechanics.
120 \details These are typically small behaviours for visible in-game objects.
121  It is in the sense of Unity to spread the responsibility for game mechanics to different scripts
122  which can run on their own, instead of off-loading everything to a single large manager.
123 
124 \dir Assets/Scripts/GameManagement/Player
125 \brief Scripts related to handling (adding, customizing, removing, syncing ...) player (not client!)
126  information. See namespace \ref player.
127 \details This information is planned to be tracked as common room property of all clients logged in.
128  Additionally there are scripts responsible for ensuring how player properties (color, input scheme,
129  etc.) can be chosen.
130 
131 \dir Assets/Scripts/GameManagement/Settings
132 \brief Holds all scripts related to editing, saving, loading and accessing the game's settings.
133  (No own namespace yet.)
134 \details Settings are wip currently.
135 
136 \dir Assets/Scripts/Helpers
137 \brief Contains common scripts and base classes usable across different concepts.
138  (No namespace restrictions for ease of use.)
139 \details If bigger collections of coherent helper classes are intended to work together, please consider
140  treating these as a framework and move them to their own Frameworks/[Topic] folder.
141  Example: StateMachine and State base implementations could be move to a SM framework.
142 
143 \dir Assets/Scripts/Net
144 \brief Scripts related to networking, i.e. connection management and data transmission schemes.
145  (See namespace \ref net.)
146 \details While the NetManager is the current-state representation of the networking feature, the NetAdapter
147  is the abstraction layer to the used networking API.
148  This way, if the API changes, not everything needs to be touched.
149 
150 \dir Assets/Scripts/Net/Events
151 \brief The NetEvent base class and definitions of network data to be used with it.
152  (Part of namespace \ref net.)
153 \details Please put new data classes for use with NetEvent here. These have to implement the INetDataPkg
154  interface.
155 
156 \dir Assets/Scripts/StateMachines
157 \brief All states and SM (Animator) definitions are collected here, sorted by SM use case.
158 \details Please put new SMs exclusively here.
159 
160 \dir Assets/Scripts/UI/Dialogs
161 \brief The scripts for modal and non-modal dialogs go here. All go to namespace \ref ui currently.
162 \details Try and make dialogs flexible, so they can be used without big overhead from different point in the
163  project like: Selection Dialog, Message Dialog, Confirmation Request etc...
164 
165 \dir Assets/Scripts/UI/MainMenu
166 \brief Contains scripts attached to objects from the main menu. They have their own namespace:
167  \ref ui::mainmenu
168 
169 \dir Assets/Scripts/UI/MenuSystem
170 \brief This includes all assets necessary for the generalized menu animation system.
171 \details The system is not very good and flexible still. We need to work on it.
172 
173 \dir Assets/Scripts/UI/Overlays
174 \brief Scripts for in-game world-coordinate UI overlays, such as floating labels etc.
175 \details The name 'Overlays' may be a bit misleading. What do you think?
176 
177 \dir Assets/Test
178 \brief This folder contains all material that is special for tests of our features.
179  Namespace \ref test.
180 \details This may include everything normally sorted to the other folders, like own test scenes, prefabs,
181  materials, scripts etc...
182  You can use this folder for playing around and testing your features, but we will maybe also need
183  some structure for our testing in the future. Do we love or hate tests, or both?
