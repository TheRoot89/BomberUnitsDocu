3 The UI Menu Page System
4 =======================
5 
6 The menu page system is a custom grouping and animation system for UI Menu content. It currently features
7 bringing individual elements in and out of view, as well as grouping these elements into pages.
8 
9 The System
10 -------------
11 Have a look at the test scene: Assets/Test/MenuAnimationWithLayoutsTest.unity
12 
13 The basis of the system is using an animation state for each UI::MenuElement. Each can be in a
14 state of 'shown' or 'hidden' to one of the four sides of the screen.<br>
15 Each "animation" only consists of one frame: the new position the object shall have. The actual visible
16 animation of an element sliding out or in is done by the smooth transition between those 1-frame-animations.
17 
18 Each item to be animated thus need the component 'Animator', as well as the script MenuElement.
19 
20 These "elements" can be grouped to form a logical "page", by listing these elements on an empty object
21 featuring the UI::MenuPage component. They do not have to be children of the page, but only be
22 listed there! This allows the same elements to be on multiple pages, so when you switch a page, only the
23 subset of removed or added elements needs to change.
24 
25 > **Note:** The pages should be children of a 'Pages' parent.
26 
27 *Elements* and *Pages* provide functions to hide or show them. Have a look at their documentation for details.
28 
29 Switching between pages
30 -----------------------
31 
32 Should you not want to implement a callback for your UI management for every possible transition and do not
33 set callbacks to buttons directly, you can set up `Triggers` on a per-page basis.
34 The UI::Trigger instances can be set for page objects in the editor, taking a button and a command.
35 Upon showing of a page, the trigger will register a callback to the button, issuing the command.
36 
37 > This is not finished yet, as the trigger searches for a main menu state machine. This has to be made more
38 > versatile to use triggers in any situation. Proposal: re-route over a "UIManager" object.
39 \todo Triggers are not final.

41 Coping with Layouts
42 -------------------
43 
44 Layouts, together with anchors, are very useful in calculating the correct sizing and positioning for all
45 objects automatically. But Layouts also prevent objects from moving, i.e. being animated with a shift, as they
46 are in control of the element's position instead of the animator (coordinates are greyed out).
47 
48 As a solution, we need to layout an invisible parent of the object-to-be-animated and shift the actual
49 element relative to its parent. The parent inherits the size of its child and is managed by the layout.
50 
51 So, to get both worlds, set the the content you want to animate as child of an empty RectTransform object.
52 This *container* will get layed out by its respective parent having a layout component. The actual
53 `MenuElement` in its shown state is set to be in relative position (0,0) and is thus the same as its
54 container. This relative position can be modified by the Animator to move off-screen, while the container
55 actually stays in place as set by the layout.
56 
57 > Have a look at the example scene for this. Here the 'Layout containers' for the elements are set, but not
58 > referenced in the 'Container' setting of the `MenuElement`. This is why the layout stays the same,
59 > even if elements are panned out and deactivated. If instead the container is set to the script, also
60 > the element would be added/removed from the layout upon activation/deactivation, changing the space
61 > available for all elements and resizing/shifting them automatically.
