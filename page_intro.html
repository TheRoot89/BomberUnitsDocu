Getting started

Alright recruit, to get you up to speed we will (probably) need five things:

  1. Do the necessary installations
  2. Clone our repo from GitHub
  3. Understand the basics of our tools
  4. Have a rough picture on the state of the project
  5. An example of a typical SW change workflow

1: Do the necessary installations

The correct Unity Version via Unity Hub
Hop on over to Unity, select the Unity Hub for download and go through the setup.
Unity Hub allows you to manage multiple installations of Unity on your computer, as well as associate your projects (and our project) with one of these installations. This is a good thing, as we need to work all with the same version but you may, for example, need want to switch back to an old branch requiring an older Unity.
Install and select the currently used Unity version from the Hub App. You can find our currently used Unity version near the top of our mainpage in the "Last stable release" note.

Visual Studio or Visual Studio Code
The Unity Editor comes with built-in support vor VS and VS Code. This means Unity will create the project configuration and start it up when you click to edit a script from your game's Assets. You need to have one of the above installed. Unity offers to install a VS version automatically with its main installation.

Toni strongly recommends though to check out VS Code as it is awesome. To switch to VS Code support from the default VS: Open up Unity Preferences, External Tools, then browse for the VS Code executable as External Script Editor. VS Code profits from a vast range of plugins to make your life easier and some especially useful for us are listed on the Tooling page: Tooling
Also look here for more info on VS Code used with Unity.

Git
Git is one of the currently most popular Versioning Tools out there. It can be used from the command line, but got a lot of graphical front ends from hobby devs and companies alike. Download it here and be sure to understand the basics of it by making use of the vast amount of tutorials on the net (for example on the Git homepage itself).

Doxygen
...is the defacto standard for automated documentation generation for all kinds of languages. This is not a must, but dearly recommended, as we try to document lots of useful stuff nicely readable there **fingers crossed**. Get it here and install. This is also a command line tool - usage instructions can be found on our project homepage mainpage.

2: Clone our repo from GitHub

Your Git installation gives you access to our repository on GitHub. You may want to download and install the GitHub Desktop client for ease of use.
If you choose to do so, you will simply have to log into the client and can already add a repository from GitHub from the "Current Repository" section at the top left -> "Add" -> "Clone Repository". You may have to give the direct ULR (as from the browser) of the repo as it is set to be private to us.

After the clone, be sure to execute
git lfs install
from your local repo folder once, using for example the Git Bash, which comes with your Git installation. This ensures large files are not saved in every version on every client, which is important when working with Unity Assets.

You will now have the newest "Master" branch checked out. The folder can be selected from Unity Hub as project and should open with an installed Unity version as given on the top of the mainpage.

5. An example of a typical SW change workflow

For a feature implementation, things may go like this:

  1. We decide together which features we want to implement next and move these on our GitHub Repo Projects page from the "Ideas" column to the "ToDo" column.
  2. Choose one of these tasks and convert the note card to a full Issue via "..." -> "Convert to Issue", if that was not already done before. You also assign yourself as the "Assignee" of that task.
  3. Fill the issue with whatever info you feel necessary or start a conversation using the comment section.
  4. Create a branch (from master most of the time), mentioning the issue number in its naming.
    Example: "feature/#4_implement_multiplayer".
    Using the "feature/" prefix allows us to sort the branch names after features, bugfixes, releases and whatever categories we want to define.
  5. Work on you branch and add the feature, using Git and pushing your changes to our repo. Best move your issue from the "ToDo" column to the "In progress" column.
  6. If there were changes on the master branch from when you branched off it, merge "master" to your branch to check compatibility, resolve merge conflicts and test whether everything still works as expected.
  7. Go to GitHub, select your branch on the left and click "New pull request". Fill in the details of what you did on your branch, assign Reviewers on the top right and click "Create pull request".
  8. Once the reviews from us approve your changes, the created pull request (PR) can be merged in GitHub. If you don't need it any longer, your branch can be deleted (this will only delete the name, not the commits you did) and you can close your Issue and move it to the "Done" column! Yeah!

On Unity

Unity is a game engine and tool collection for this engine. There is no real winner when comparing Unity to other free engines, like Unreal - so we just start with the one that looks easier to pick up.

Unity comes with its own graphical editor to arrange your game world in what Unity calls Scenes. In-game you as a player will always be in one of these Scenes. It may and will feature so-called GameObjects, representing any kind of entity present int the Scene, e.g. a player figure, bullets, particle effects or invisible objects only there to hold Scripts.

Scripts represented the logic of a game, implemented in C#. These can be written as in any other project, defining and creating objects and calling the Unity API to act on the game world.

Interaction of your Scripts with the engine comes in two directions:

  • Script to engine: Your Script causes something to happen by calling the API, e.g. to change the position of a game object.
  • Engine to Script: Some predefined Script functions can be called from the game and are thus called Callbacks. An example is the Update() function called by the game for every frame.
    In order for this to work your Script's class has to be a Behavior, by inheriting from the MonoBehavior class. In order for your class to be instantiated and the Callbacks (like Update) to be called, it also has to be added to a GameObject in the Unity editor. That GameObject will now have the additional Behavior you code for it.

Example: A PlayerController script is often added to the player's figure in the game world. Each frame its Update function will be called by the engine. Within that function, code may be added to check the input keys a player may be pressing to control his figure and the figure may then be moved accordingly.

About the game

So, Bomberman is a classic game type (Google!). We first want to replicate its classic, simple scheme of moving your player figure, lay bombs, collect PowerUps and finally explode your opponents.

As this is vitally important today, we also want to support mixed multiplayer (local, online). Of course a game does not just start directly with a level and your fellow players already there. There will have to be a menu UI, (online) player management, level selection, game-over screen etc...

When this runs stable, we want to add our own take on the genre and add our own features. This even more so than the above points is a group effort, where we can brainstorm, decide, implement and test together.

So how can I contribute?

The are quite a lot of things to be done, where you can pick what you think is most fun for you. A brainstormed list, with the must-haves in bold:

  • Game Design
  • Planning & Teamwork
  • Art Design & Modelling
  • Rigging & Animation
  • Level Design
  • Scripting
  • Effects
  • Post processing
  • DevOps
  • Testing

Okay, what does the above mean, roughly, and how would I go on about it?

Game Design

The core of any project: figuring out what the f*** we actually want. In order to get going, we need at least some basic ideas, preferably with an interesting twist. Then we need to refine these ideas to answer questions such as:

  • How will playing this be fun?
  • What are the core elements and themes?
  • What shall the Player's experience be like?
  • What should the graphical style be like? What that of the UI?
  • What main functionality will the SW have to provide?
  • What other, crazy ideas do we have? Do they fit in?

The tools to support this are up to discussion - I just started with plain Powerpoint.

Planning & Teamwork

So, imagine we have quite a big list of tasks, big and small, crazy ideas and stuff we just want to try to see whether they work. We need to:

  1. Decide what has to be done in which order from a technical perspective.
  2. What we want to do first of the things that require no special order. What are the ratios of gain vs. effort for the features we want to add?
  3. Who wants to do what? Is everyone happy with what he is doing and where we are going? Does anyone need help?

Also, stuff needs to be taken care of:

  • When and how do we meet?
  • How do we want to collaborate? Are there any rules?
  • Most importantly: How can we improve?

One tool to support this is surely the project tab and issue tracking of GitHub, where our code is hosted anyway. It offers a way to collect, sort and prioritize ideas as notes, track tasks, and rant about them in strong language. Information not related to ideas or tasks may be tracked using these nice Markdown pages.

Art Design & Modelling

If this is not a text adventure, we will need to show some art, be it 2D or 3D. How can we achieve this without hiring Weta Workshop? How do we achieve a reasonable look with comparably low effort?

The tools to support this are manyfold, with many specialized solutions for the different artsy tasks, depending on what we want. Simple models can be made for example with AssetForge, while arbitrarily complex stuff can be done with Blender. Maps and textures can be found, also free, all over the web. The final look will also largely depend on the Postprocessing Stack, allowing cool looks for low-poly models.

Rigging & Animation

Rigging is the task of providing a virtual skeleton, or rig, for your models. This allows to deform the models to assume natural looking poses. A sequence of such poses forms an Animation, which is a kind of asset importable and replay-able in Unity.

Tools to support this: Rigging is typically supported by modeling SW, such as Blender. Animations may be done in the Unity editor as well, while the final chaining of animations with animation state machines is a task of game development, also supported graphically by the Unity Editor.

Level Design

This is what the title implies, and you have probably seen good and bad levels abound yourselves. It is a bit of a cross-over task:
The design aspect of this is a part of the Game Design, especially when thinking of cool level features, which will need support from a Scripting perspective.

For the actual modeling of a level Scene, using mainly the Unity Editor as Tool, you will combine Models as well as maybe create your own stuff, as a Modeling guy would do. Poor fella.

Scripting

This is the art of learning what the Unity API has to offer, how to use that to design an understandable and reusable SW architecture and - of course - coding the thing in C# (or Java Script alternatively). The scripts bring the game objects to life, let you play with friends, produce tons of log output, and so on...
Game development may sound like it is all about Coding/Scripting, but as you see, there are many daunting tasks each worthy ofa whole profession in the industry.

The tools to do that are Unity and a supported IDE, such as Visual Studio or Visual Studio Code. Have a look at the Tooling section (Tooling) for how to set those up. There is no black magic involved, as Unity pretty much has the whole toolchain, up to deployment, built-in. Nice.

Effects

This category wraps up some of the typical features found in Game Engines: particle systems, sound effects and the relevant scripting making effects affecting the game world. This is also a bit of a crossover task, and you get to play with explosions. Also in the effects category are screen effects, supported by the rendering pipeline used, which is quite the topic of its own.

The Unity editor may be all the tool you need to get started. Also Unity gives away quite a lot of free-to-use effects we can start with. Building on that, you may need to roam the Net and use some tools of the other disciplines to create you very own uber-effects!

Post processing

In Post processing, meaning after the basic rendering of the frame, it is possible to apply further transformations to apply effect or improve style and image quality. Typical tasks are Anti-Aliasing or the Bloom effect. This has to be designed, tuned and tested. From the tooling perspective, all this should be supported by Unity itself I think - but I am by no means experienced in that area.

DevOps

If we decide to have, e.g. build automation, common tool configurations (such as for Doxygen in the doc folder), etc. - someone will have to look after that. Someone willing to be the all-important enabler of the whole team. That someone could be you!

The skills learned/needed here are probably no special to game development, and the tools are manyfold, depending on what the team deems worthy automating.

Testing

No time for the other stuff? No worries - we need people that from time to time have a look at the current state and tell us what we f**** up. At pat on the back is also helpful sometimes.

And then, there is proper testing, including Unit Tests. It may not seem reasonable for the dynamic nature of game scripting code, but it is very well possible and supported by Unity's own set own tools. The scripters will kiss the feet of the guy who tells them how to use those tools. At the very latest when complexity increases and devs become afraid of breaking things accidentally.

Where to learn more?

Luckily, there is an abundance of resources on Unity (and probably on the other stuff as well) on the web. You may start with the tutorials present on the Unity website. Or you can take courses on leaning portals, such as Zenva or Udemy. The best way to learn of course is to take part in this project and learn from and with your friends. Most importantly, don't think anyone will look down on you because you are just learning a new thing - this is not Dota after all... (yet).
