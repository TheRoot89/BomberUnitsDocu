Management

The Management Layer (click for details) is responsible for various basic, background tasks. The following important managers exist, with each topic being fairly independent. Have a look at their detailed design docu:

Component Short description
Management::GameManager Manages the overall game state: startup, loading, quit
Management::UIManager Manages UI display and inputs and delegates reactions accordingly
Management::MatchManager Initializes and manages a single match's state
Management::PlayerManager Manages local and remote players
Management::NetManager Manages Network interactions with Photon for enabling MP

Most important is the GameManager, featuring its own state machine:

  • Concept: Game Management and Control Flow
  • Concept: Handling State
  • State Machine: Management::GameStateMachine
  • Also see: The Game Startup : How the components work together during startup

Additionally, there is a range of custom classes which don't need to be behaviors. These are often data structures, such as:

  • Management::Player
  • Management::InputScheme
  • Management::GameSettings
  • Management::Level

UI

The UI implementations one the one hand need to provide functionality to compose, animate and switch menu pages.
Architecture Details: The Menu Page System
Concept page: Menu page System

Component Short description
UI::MenuPage Groups MenuElements to be shown together as a menu page
UI::MenuElement Added to game objects to enable their animation
UI::Trigger Provides Button Callback registering at creation
UI::MainMenuStateMachine Treats each main menu page as a state

On the other hand there are single-use pop-ups (like simple menus without transitions). These will be called Dialogs, but this is work in progress.

And finally additional UI content, such as in-game hovering labels and other effects:

  • UI::Label
  • UI::Billboard

Mechanics

Game Mechanics represent core game design ideas, spread out over various scripts. Some of these provide input management (Mechanics::PlayerController), while other provide behaviours for game objects:

  • Mechanics::Breakable : Exposes the Break() to replace a model by its broken variant.
  • Mechanics::Explodeable : Exposes the Light() function to trigger a delayed explosion of the object.
  • Mechanics::LootSpawner : Exposes the RollDropLoot() function
