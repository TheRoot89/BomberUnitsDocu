SW Architecture

These pages shall give an overview on how the overall system works. The SW architecture realizes the game concepts (See Concepts) in a hopefully understandable way. Below the architecture come the individual SW components, which are documented by code comments, generated into the API.

Note: The current SW is not very structured and, actually, not very big. The architecture needs to be split in more domains, like UI and Behaviours etc, reflected by namespaces in the actual SW and Boxes around the classes is pictures here. There is a ticket for it...

The Game Management Layer is responsible for various basic, background tasks. Its components are implemented as behaviours and attached to invisible management GameObjects named after them.
Those components which track their state have an additional state machine script attached to the same object.

Please find information on the individual managers in their source comments or the detailed description section in their Doxygen docu here:

Component Short description
BomberUnits::GameManager Manages the overall game state: startup, loading, quit
BomberUnits::UIManager Manages UI display and inputs and delegates reactions accordingly
BomberUnits::MatchManager Initializes and manages a single match's state
BomberUnits::PlayerManager Manages local and remote players
BomberUnits::NetManager Manages Network interactions with Photon for enabling MP

Custom Classes

Additionally, there is a range of custom classes which don't need to be behaviors. Most importantly:

  • Container types, such as:
    • BomberUnits::Player
    • BomberUnits::InputScheme
    • BomberUnits::GameSettings
    • BomberUnits::Level
  • UI classes for managing UI components, grouping and animation. See:
    • BomberUnits::UIMenuPage
    • BomberUnits::UIMenuElement
    • BomberUnits::UITrigger

Other Scripts

Categorized into folders we also have scripts for:

  • Game Mechanics:
    • BomberUnits::PlayerController : manages player interaction with the player figure
  • Attributes, which provide further functionality/interactions to GameObjects. Each attribute script can be configured from the editor, or by loading from the settings.
    • BomberUnits::Breakable : Exposes the Break() to replace a model by its broken variant.
    • BomberUnits::Explodeable : Exposes the Light() function to trigger a delayed explosion of the object.
    • BomberUnits::LootSpawner : Exposes the DropLoot() function
  • UI Behaviours, providing UI features for objects they are attached to:
    • BomberUnits::UILabel
  • Misc, for other topics, such reading and writing from/to files.
  • State Machines, which are associated to certain objects:
    • BomberUnits::GameStateMachine
    • BomberUnits::MainMenuStateMachine
