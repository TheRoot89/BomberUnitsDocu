UI State System

The UI state system

The UIManager can be seen as being in different states, depending on what screen the user is on. Often a state is associated with a page from the Menu page System, but this is not necessarily so.

The state pattern

We use an instance of the State pattern here, where each state is represented by its own class. All these concrete states inherit their methods from a GameState base class, and override these to provide individual handling of a request depending on the current state. The GameState can call its state type (GameState) to handle a certain string command. It does however not know itself which concrete state it is in, so this concept is encapsulated. The non-abstract implementation of this handler routes the request to the correct virtual method, which has a default (error-)implementation that has to be overwritten and specified by the concrete state classes. So the state object decides on its own what is to be done about the request and whether to change the state as a consequence. This is an application of the Template Method pattern.
